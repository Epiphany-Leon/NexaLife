# 真实邮箱账户体系与多设备登录设计

日期：2026-03-13

## 先说结论

当前 NexaLife 的邮箱验证码流程只能算“本机邮箱校验”：

- 账户信息保存在本地 `UserDefaults`
- 验证码由客户端本地生成
- 登录状态也只存在当前设备

所以它不能支持真正的：

- 跨设备登录
- 服务端找回账户
- 多端共享同一份数据
- 版本更新通知与设备消息推送

要做到真正的邮箱账户体系，必须把“身份、会话、同步”三件事都迁到服务端。

## 推荐的总体架构

拆成四层：

1. `Auth Service`
   
   - 负责邮箱验证码发送、验证码校验、账户创建、登录、刷新 Token。

2. `Account Service`
   
   - 负责用户资料、通知订阅偏好、设备列表、Apple ID / 邮箱身份绑定。

3. `Sync Service`
   
   - 负责多设备之间的数据上传、拉取、冲突处理。

4. `Push / Notification Service`
   
   - 负责版本更新通知、系统消息、营销通知、设备 Push Token 管理。

## 最小服务端数据模型

至少需要这些表：

### `users`

- `id`
- `display_name`
- `primary_email`
- `email_verified_at`
- `created_at`
- `updated_at`

### `user_identities`

- `id`
- `user_id`
- `provider`
  - `email`
  - `apple`
- `provider_subject`
  - 邮箱可存标准化邮箱地址
  - Apple 可存 `credential.user`
- `verified_at`
- `created_at`

### `email_verification_codes`

- `id`
- `email`
- `purpose`
  - `sign_up`
  - `sign_in`
  - `bind_email`
- `code_hash`
- `expires_at`
- `consumed_at`
- `created_at`

注意：

- 不要明文存验证码
- 只存哈希
- 校验成功后立即标记 `consumed_at`

### `sessions`

- `id`
- `user_id`
- `refresh_token_hash`
- `device_id`
- `client_platform`
- `expires_at`
- `revoked_at`
- `created_at`

### `devices`

- `id`
- `user_id`
- `device_name`
- `platform`
- `push_token`
- `last_seen_at`
- `created_at`

## 真正的邮箱登录流程

### 1. 发送验证码

客户端输入邮箱后调用：

- `POST /auth/email/send-code`

服务端做这些事：

- 标准化邮箱地址
- 限流
- 生成 6 位验证码
- 保存哈希后的验证码
- 调用邮件服务商发送邮件

### 2. 校验验证码并创建/登录

客户端输入验证码后调用：

- `POST /auth/email/verify-code`

服务端做这些事：

- 校验邮箱和验证码
- 若用户不存在则创建用户
- 若用户已存在则直接登录
- 签发：
  - `access_token`
  - `refresh_token`
  - `user`

客户端做这些事：

- 把 `access_token` 和 `refresh_token` 存进 Keychain
- 把 `user.id` 作为本地账号主键
- 本地 `AppState` 不再自己生成邮箱账户 ID

### 3. 自动续期

客户端在 Token 过期前调用：

- `POST /auth/refresh`

服务端返回新的 `access_token`。

### 4. 登出

客户端调用：

- `POST /auth/logout`

服务端撤销当前 session。

## 多设备登录怎么做

关键不是“验证码能在多台机器上收”，而是“多个设备拿到的是同一个服务端用户 ID”。

正确做法：

1. 第一台设备登录邮箱，服务端创建 `user_123`。
2. 第二台设备再用同一邮箱登录，服务端仍返回 `user_123`。
3. 两台设备都拿到各自的 session，但绑定到同一个 `user_id`。
4. 之后数据同步都按 `user_id` 归属，而不是按设备归属。

## 多设备同步推荐分两阶段

### 阶段一：先做“账户 + 云端快照同步”

这是最省工程量的版本，适合先跑通。

服务端增加一张表或一个对象存储：

- `user_snapshots`
  - `user_id`
  - `snapshot_json`
  - `latest_change_at`
  - `updated_at`

流程：

1. 客户端登录后带着 `access_token` 请求自己的快照。
2. 本地有更新时，把整份 JSON 快照上传到服务端。
3. 新设备登录后拉取服务端最新快照并覆盖本地。

优点：

- 开发快
- 可以复用你现在已有的 `NexaLifeDataArchive`

缺点：

- 冲突粗糙
- 最终还是“整包覆盖”
- 不适合频繁多端同时编辑

### 阶段二：升级到“记录级增量同步”

这是长期正确方案。

每个业务模型都要有：

- `id`
- `userID`
- `updatedAt`
- `deletedAt`
- `syncVersion`

同步接口改成：

- `GET /sync/changes?cursor=...`
- `POST /sync/push`

流程：

1. 客户端记录本地变更。
2. 推送本地变更到服务端。
3. 服务端返回已接受版本号。
4. 客户端再拉取自上次游标之后的远端变更。
5. 按记录合并到本地 SwiftData。

## 冲突策略

初期建议：

- 统一用 `updatedAt` 最后写入胜出

再下一步才升级为：

- 文本类字段做字段级合并
- 删除操作用 tombstone
- 某些模块单独做冲突提示

## 客户端要改什么

NexaLife 客户端至少需要改这几件事：

1. `OAuthService`
   
   - 不再本地保存邮箱账户
   - 改为调用真正的 Auth API

2. `AppState`
   
   - `accountIdentifier` 改为服务端 `user.id`
   - `authToken` 不再等于本地生成的 UUID

3. 凭证存储
   
   - `access_token`
   - `refresh_token`
   - 全部进 Keychain

4. `AppDataArchiveService`
   
   - 阶段一可继续作为快照同步载体
   - 阶段二拆成增量同步服务

5. 每个模型补齐同步字段
   
   - `updatedAt`
   - `deletedAt`
   - `userID`
   - `syncVersion`

## Apple ID 和邮箱如何共存

正确做法不是做两套账户，而是一套 `users`，多套 `identities`。

例子：

- `users.id = user_123`
- `user_identities`
  - `provider = email`, `provider_subject = leon@example.com`
  - `provider = apple`, `provider_subject = 001234.abcd...`

这样同一个人可以：

- 先用邮箱注册
- 之后绑定 Apple ID
- 或者反过来

## 推荐开发顺序

1. 先把当前本地邮箱验证码流程换成服务端验证码流程。
2. 引入 `access_token + refresh_token + user.id`。
3. 做服务端“整包快照同步”。
4. 新设备登录后自动拉取快照恢复数据。
5. 稳定后再升级为记录级增量同步。

## 直接给你的建议

如果你的目标是尽快上线可用版本，不要一开始就做全量记录级同步。

优先做这条路线：

1. 邮箱验证码服务端化
2. Session / Token 服务端化
3. 复用当前 JSON 快照做云端同步
4. 跑通多设备登录与数据恢复
5. 之后再升级成真正的增量同步

这是当前成本最低、也最符合你现有代码结构的路径。
