# NexaLife 数据同步与账户设计

日期：2026-03-13

## 当前版本落地内容

1. 账户体系
   
   - Apple ID：改为原生 Apple 授权入口，不再使用无效的 Web OAuth 占位地址。
   - 邮箱账户：支持任意邮箱验证码创建与登录，已验证邮箱可作为后续版本更新和重要通知触达渠道。
   - 本地导入：首次启动时可直接导入导出的 NexaLife JSON 数据包。

2. 数据导入导出
   
   - 统一改为“应用级 JSON 快照”。
   - 快照包含：
     - 所有 SwiftData 模型数据
     - 账户资料
     - 偏好设置
     - 头像图片内容
   - 不包含：
     - API Key
     - 本机路径类配置

3. 文件夹同步
   
   - 同步目录内维护一个固定文件 `NexaLife-sync.json`。
   - 同步策略采用最新快照覆盖旧快照。
   - 当前适合单人、单账号、低冲突场景。

## 同步逻辑

1. 本机生成当前数据快照，并附带 `latestChangeAt`。
2. 若同步目录不存在远端快照，则直接推送本机快照。
3. 若远端 `latestChangeAt` 更新，则拉取远端覆盖本机。
4. 若本机 `latestChangeAt` 更新，则用本机覆盖远端。
5. 时间戳相同则视为无变化。

## 为什么不用直接复制 SwiftData store

1. 当前运行中的应用持有活动中的 ModelContext，直接替换底层 store 容易造成导入失败或状态污染。
2. JSON 快照更适合：
   - 清空内容
   - 导入本地数据
   - 手动同步
   - 自动备份

## 后续建议

1. App Store / iOS 正式版
   
   - Apple ID + CloudKit 作为官方同步主路径。
   - 继续保留当前 JSON 快照导入导出，作为迁移与灾备手段。

2. 开源 / 自托管路线
   
   - 在当前快照层之上增加远端对象存储或 WebDAV 适配器。
   - 保持快照协议不变，只替换传输介质。

3. 冲突处理升级
   
   - 当前为最后写入胜出。
   - 下一阶段可为每个模型增加 `updatedAt`，升级到记录级合并。
