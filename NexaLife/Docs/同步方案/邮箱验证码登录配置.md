# 邮箱验证码登录配置

NexaLife 当前的邮箱登录客户端已经接好验证码流程，但真正发送邮件还需要一个服务端接口。

## 当前客户端约定

- 环境变量 `NEXALIFE_EMAIL_VERIFICATION_ENDPOINT`
  - 值：负责发送验证码邮件的 HTTPS 接口地址
- 环境变量 `NEXALIFE_EMAIL_VERIFICATION_KEY`
  - 值：客户端请求该接口时使用的 Bearer Token

如果这两个环境变量没有配置，NexaLife 会进入开发预览模式：

- 不真正发邮件
- 在引导页直接显示当前验证码，方便本地联调

## 客户端会发送什么

当用户点击“发送验证码”时，NexaLife 会向 `NEXALIFE_EMAIL_VERIFICATION_ENDPOINT` 发起 `POST` 请求：

```http
Authorization: Bearer <NEXALIFE_EMAIL_VERIFICATION_KEY>
Content-Type: application/json
```

请求体 JSON：

```json
{
  "email": "user@example.com",
  "code": "123456",
  "purpose": "createAccount",
  "displayName": "Leon",
  "expiresAt": "2026-03-13T01:23:45Z",
  "app": "NexaLife"
}
```

其中：

- `purpose` 取值为 `createAccount` 或 `signIn`
- `code` 由客户端生成，当前有效期为 10 分钟
- 服务端只需要完成邮件投递并返回 2xx

## 服务端最小职责

服务端至少需要做这几件事：

1. 校验 `Authorization` 头是否正确。
2. 解析 JSON 请求体。
3. 将 `code` 和 `expiresAt` 写入邮件正文并发送到 `email`。
4. 成功时返回 `200-299`，失败时返回非 2xx。

注意：

- 当前版本的验证码校验状态保存在本地 `UserDefaults`，所以服务端不需要存验证码。
- 如果后续要做真正的跨设备邮箱账户体系，就需要把账户、验证码、会话都迁移到服务端。

## 推荐接入方式

- 开发联调：继续使用当前预览码模式，不配环境变量。
- 小规模真实发送：加一个很薄的邮件转发接口，后面接 Resend / Postmark / SES 任一邮件服务。
- 正式上线：把账户注册、验证码签发、验证码校验、消息订阅偏好都放到服务端，不再由客户端本地持有。

## Xcode 本地配置

如果你要在 Xcode 里本地联调真实邮件发送：

1. 打开 `Product > Scheme > Edit Scheme...`
2. 选择 `Run > Arguments`
3. 在 `Environment Variables` 中添加：
   - `NEXALIFE_EMAIL_VERIFICATION_ENDPOINT`
   - `NEXALIFE_EMAIL_VERIFICATION_KEY`
4. 重新启动 App
